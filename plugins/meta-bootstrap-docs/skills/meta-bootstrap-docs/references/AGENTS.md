# AGENTS.md 設計原則摘要

源自 Augment Code 對大型 monorepo 中數十份 AGENTS.md 的實測結論。
本檔為操作時對照用的精煉版（原始文章為「AGENTS.md 設計實驗」一文）。

## 核心數據（為什麼這件事值得認真做）
- 最佳 AGENTS.md ≈ 把模型從 Haiku 升到 Opus 的效果
- 最差 AGENTS.md 比完全不寫還糟
- 同一份文件在不同任務上波動可達 ±30%

提升的不是模型推理力，是「消除歧義、讓 Agent 在你倉庫裡少走錯路」的能力。

## 7 個有效模式

### 1. 漸進式披露
- 主文件 100~150 行
- 細節推到按需加載的參考文件
- 超過 150 行收益反向（中間迷失現象 Lost in the Middle）
- 150 不是真理，是當前模型注意力機制的甜區。原則是「把規則壓進注意力最聚焦的那一段」

### 2. 程序性工作流
編號式多步驟工作流是測量到最強的模式之一。
- 實例：六步部署工作流 → PR 缺檔率 40%→10%、正確性 +25%、完整性 +20%
- 本質：消除歧義，讓 Agent 不用猜

### 3. 決策表
強制提前選好「兩三種都說得通的做法」（React Query vs Zustand）。
讀完就知道走哪條路，相關 PR 最佳實踐遵從度 +25%。

### 4. 真實代碼片段（非伪代碼）
- 3~10 行短片段
- 必須來自實際生產代碼
- 超過量會引起在錯誤位置做模式匹配
- 維護成本高，**優先用 file:line 引用而非複製**；複製進來的片段必須進 Code Review checklist

### 5. 每個「不要做」配「要做」
只有警告的文檔表現一貫不如「禁令 + 替代方案」配對。
- ❌ 「不要直接實例化 HTTP 客戶端」
- ✅ 「不要直接實例化。使用 lib/http 中的 apiClient（含重試中間件）」

> 15 條以上連續禁令且沒配「要做」 → Agent 過度探索、保守行事、完成更少工作。

### 6. 文檔模組化
表現最好的 AGENTS.md 描述相對隔離的子模組。
> 一個聚焦的 150 行 AGENTS.md，放在 50 萬字符的周圍規格之上，救不了你。修復文檔環境，不是只修入口。

### 7. 領域規則具體可執行
語言特性、框架約定、安全限制 — 具體且可執行才有效。堆幾十條模糊規則等於沒寫。

## 2 個失敗模式

### A. 過度探索陷阱（上下文腐爛）
- 太多架構概述 → Agent 被拉去讀數十份文件，加載幾萬到幾十萬 token 後輸出反而變差
- 過多警告堆疊 → 30~50 條警告會讓 Agent 對每條都去驗證當前方案是否違規，即使與任務無關

### B. 描述未落地的模式
寫了「打算引入但還沒實作」的模式 → Agent 會積極地朝錯誤方向走，按文檔去找根本不存在的代碼。

> **AGENTS.md 描述現狀，不描述願景。**

## 文檔發現率（Augment Code 在數百 session 追蹤）
- AGENTS.md：100%（自動發現）
- AGENTS.md 中引用的參考文件：>90%（按需加載）
- 當前工作目錄 README.md：>80%
- 嵌套子目錄 README：~40%
- _docs/ 中無引用的孤立文檔：**<10%**

> AGENTS.md 是唯一具有可靠發現性的文檔位置。內容要被看到，要麼放這裡，要麼從這裡引用。**把內容移到被引用的位置，往往比寫更多文檔更有杠桿效果。**

## 與 README 的分工
- README 給人類看，講架構、願景、來龍去脈
- AGENTS.md 給機器看，只關心規則、工作流、防坑指南
- 兩份都要維護是真實成本

## 維護成本（誠實面對）
「使用真實代碼片段」+「描述現狀不描述願景」加起來意味著 AGENTS.md 是**高頻更新文檔**。
應對：
- 優先引用不用複製
- 複製片段進 Code Review checklist
- 定期 prune 過時警告與工作流，比加新條目更重要
